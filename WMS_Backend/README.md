# WMS 倉儲管理系統 - 開發與架構手冊

## 1. 專案簡介
本專案為一個基於 **FastAPI** 與 **SQLite** 的倉儲管理系統 (WMS)。主要用於管理企業內部的資產借用與耗材領用。
本系統不僅具備基礎的借用管理，更導入了**企業級稽核與防呆機制**，包含：自動化庫存鎖定（防超貸）、主管雙軌審核、退件逾期失效、實體部分毀損連動扣庫存、以及確保稽核軌跡的「軟刪除」與「歷史快照」架構。

### 🌟 v1.1 最新更新與核心里程碑
* **資料庫防禦升級**：導入「軟刪除 (Soft Delete)」取代物理刪除，並在訂單建立時寫入「歷史快照 (Snapshot)」，確保歷史稽核軌跡不因人員調職或物品改名而失真。
* **流程防呆極限化**：實作動態「防超貸」庫存驗證、退件逾期自動轉「已失效」，並嚴格限制「借用中」的物品禁止由使用者自行取消，必須走實體驗收。
* **歸還與毀損連動**：新增「管理員統體驗收」API，支援提報「部分毀損」，並會自動連動扣除物品庫存表的物理總數。
* **高階邏輯報表**：完善 `View_Usage_Records` 檢視表，透過 `LEFT JOIN` 動態關聯審核主管資訊，並精準處理 `CASE WHEN` 的運算順序，自動評估使用者的歸還表現（準時、提前、逾期）。
* **自動化測試工程**：建構完整的 E2E 整合測試腳本 (`test_integration.py`)，涵蓋正/反向邊界測試，並內建暫停斷點以利資料庫即時觀測。

* **語言：** Python 3.10+
* **框架：** FastAPI
* **資料庫 ORM：** SQLAlchemy (搭配 SQLite)

## 3. 專案目錄與模組互動架構
本專案採用模組化 (Modular) 與分層架構，確保程式碼具備高維護性與單一職責。以下是各個 `.py` 檔案的詳細用途與互動關係：

```text
WMS_Backend/
├── main.py              # 啟動進入點
├── database.py          # 資料庫連線工廠
├── models.py            # 資料庫實體定義 (SQLAlchemy)
├── schemas.py           # 資料驗證模型 (Pydantic)
├── init_db.py           # 資料庫建表與檢視表 (View) 初始化腳本
├── reset_data.py        # 測試用輔助腳本 (一鍵清空資料)
├── WMS.db               # SQLite 資料庫實體檔案 (動態生成)
├── test_integration.py  # E2E 整合測試腳本 (含檢視斷點)
├── routers/             # API 路由層 (Controllers)
│   ├── __init__.py      
│   ├── users.py         # 人員管理 (包含軟刪除)
│   ├── items.py         # 物品管理 (包含軟刪除)
│   ├── records.py       # 核心借用與生命週期狀態機
│   └── dashboards.py    # 唯讀報表 (直連 View)
└── services/            # 外部服務整合層
    ├── __init__.py      
    └── notifications.py # 模擬通知服務 (Line/Email)
```

### 檔案職責與互動說明
* **`main.py` (總機)**：系統的啟動入口。負責建立 FastAPI 實例，並將 `routers/` 資料夾下的所有 API 路由 (如 `/api/users`, `/api/records`) 掛載（`include_router`）進來。
* **`database.py` (連線工廠)**：設定 SQLAlchemy 引擎與 `SessionLocal`。它提供了一個 `get_db()` 依賴注入函數，讓所有 API 都能安全地取得與關閉資料庫連線。
* **`models.py` (底層藍圖)**：定義了 `Users`, `Items`, `Records` 的 Python 類別，直接對應到資料庫內的表格結構與 Foreign Key 關聯。
* **`schemas.py` (安檢門)**：定義前端傳來 (Request) 與後端吐出 (Response) 的 JSON 格式。攔截不合法的資料（如數量小於 0、缺少必填欄位）。
* **`routers/records.py` (核心大腦)**：系統中最複雜的模組。它會：
    1. 透過 `schemas.py` 驗證前端資料。
    2. 呼叫 `database.py` 取得連線。
    3. 查詢 `models.py` 取得物件，並透過執行 SQL 查詢 `View_Item_Inventory` 進行防超貸檢查。
    4. 將「歷史快照」寫入訂單，並更新狀態機。
* **`init_db.py` (基礎建設)**：負責執行純 SQL 語法。當資料庫不存在或結構大改時，由它負責建立具有 `is_active` (軟刪除)、`damaged_qty` (毀損數) 的資料表，以及兩張報表專用的邏輯檢視表 (Views)。
* **`reset_data.py` (清道夫)**：開發與測試的好幫手。會依照 Foreign Key 限制的安全順序（先刪 Records，再刪 Users/Items），將所有表格資料清空，但不破壞 `init_db.py` 建立好的表格與 View 架構。

## 4. 核心業務邏輯 (Business Logic)

### A. 借用與防超貸機制
* 依據物品屬性自動決定流程：耗材免還直接結案；資產依據設定進入「待審核」或「已預約」。
* **強制歸還時間**：只要是借用「資產」，系統強制要求填寫「預計歸還時間」，耗材則免填。
* **精準防超貸**：訂單建立前，系統會比對 `View_Item_Inventory` 中的「實際可用」數量，庫存不足即阻擋 (400 Error)。

### B. 狀態機與生命週期
* **雙軌退件機制**：
    * **退回修改 (Soft Reject)**：保留凍結庫存。若超過「預計取用時間」仍未重新送審，狀態自動判定為「已失效」並釋放庫存。
    * **直接駁回 (Hard Reject)**：狀態轉為「已駁回」，立即釋放庫存並觸發通知。
* **取消申請**：使用者只能在實體發放前（待審核、已預約、退回修改）自行取消，一旦進入「借用中」則必須透過管理員走實體歸還流程。
* **管理員統一點交**：無論資產是否需審核，實體的「發放 (Pickup)」與「歸還 (Return)」一律限制只有 `role = '管理員'` 的人員可執行。

### C. 稽核軌跡：軟刪除與歷史快照
* **軟刪除 (Soft Delete)**：人員離職或物品報廢時，API 僅會將 `is_active` 設為 `0`。防止破壞歷史紀錄的關聯，同時禁止停用的人/物發起新訂單。
* **歷史快照 (Snapshot)**：借用當下，系統會將人員部門、物品名稱等資訊「影印」一份存入 `Records` 表（`snap_` 欄位）。未來的歷程報表完全依賴快照，確保跨部門成本分攤的稽核準確性。
* **部分毀損連動**：歸還時若提報毀損數量，系統會將訂單標記為「已歸還(部分毀損)」，並自動將毀損數量累加至 `Items` 表的 `damaged_qty`，從總庫存中永久扣除。

## 5. 資料庫邏輯檢視表 (Views)
本系統高度依賴檢視表來即時運算報表，確保資料的一致性：

1. **`View_Item_Inventory` (庫存動態表)**
   * 自動過濾已被軟刪除 (`is_active=0`) 的物品。
   * **實際可用 = 物理總數 - 累積毀損數量 - 借用數量 - 凍結數量**。
2. **`View_Usage_Records` (使用歷程與表現表)**
   * 直接讀取 Records 表內的 `snap_` 歷史快照，確保借用人與物品歷史名稱正確。
   * **多表關聯 (LEFT JOIN)**：透過連結 `Users` 表，動態抓取該筆訂單「審核人員」的最新姓名與部門資訊。
   * 內建複雜的 `CASE WHEN` 邏輯，精準評估歸還表現（包含：準時/提前歸還、逾期未還、逾期歸還等），並能正確處理部分毀損的時間評估。所有顯示欄位皆已全面中文化排序。

## 6. 快速啟動與測試指南 (Quick Start)

### 步驟 1：環境準備
確保已安裝 Python 3.10+，並安裝所需套件：
```bash
pip install fastapi uvicorn sqlalchemy pydantic pytest httpx
```

### 步驟 2：初始化資料庫
第一次啟動，或當 `models.py` 結構有重大變更時，請先刪除舊的 `WMS.db`，然後執行：
```bash
python init_db.py
```
*(系統將自動建立所有資料表、防呆約束與邏輯 View)*

### 步驟 3：啟動伺服器
```bash
uvicorn main:app --reload
```
開啟瀏覽器前往 `http://127.0.0.1:8000/docs` 即可使用 Swagger UI 進行視覺化 API 測試。

### 步驟 4：執行斷點整合測試 (Integration Testing)
本專案提供完整的 E2E 測試腳本，搭配暫停機制方便於 DBeaver 等資料庫軟體中即時觀察狀態流轉：
```bash
pytest test_integration.py -s
```
*(注意：務必加上 `-s` 參數，測試腳本才會在關鍵節點暫停並等待終端機按下 Enter 鍵)*

### 步驟 5：測試資料重置
若在測試過程中需要將系統還原為「空資料」狀態，請確保終端機無執行伺服器，然後輸入：
```bash
python reset_data.py
```
*(此操作僅清空資料，不會破壞資料庫架構)*
